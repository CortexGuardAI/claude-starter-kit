# Database Migrations Skill

Safe, reversible, zero-downtime-friendly database migration patterns for production systems.

## When to Activate

- Creating new tables or modifying existing ones
- Adding or removing columns, indexes, or constraints
- Planning schema changes for zero-downtime deployments
- Writing ORM migrations (Prisma, Drizzle, Alembic, GORM, TypeORM)
- Backfilling data in existing tables
- Planning rollbacks

---

## Core Rules

1. **Always create migration files** -- Never modify schema directly in production
2. **Every migration has a rollback** -- `up` and `down` functions always
3. **Test on staging first** -- Never run on production without staging validation
4. **Ask before executing** -- Always confirm with the team before running migrations
5. **One concern per migration** -- Don't bundle unrelated schema changes

## Safe Migration Patterns

### Adding a Column (Safe)
```sql
-- Adding nullable column: instant, no lock
ALTER TABLE users ADD COLUMN bio TEXT;

-- Adding NOT NULL with default: safe for Postgres 11+
ALTER TABLE users ADD COLUMN is_verified BOOLEAN NOT NULL DEFAULT false;
```

### Adding a NOT NULL Column to a Large Table (Zero-Downtime)
```sql
-- Step 1: Add as nullable (instant)
ALTER TABLE orders ADD COLUMN discount_pct NUMERIC(5,2);

-- Step 2: Backfill in batches (run with app live)
DO $$
DECLARE
  batch_size INT := 1000;
  updated INT;
BEGIN
  LOOP
    UPDATE orders
    SET discount_pct = 0
    WHERE discount_pct IS NULL
    AND id IN (
      SELECT id FROM orders WHERE discount_pct IS NULL LIMIT batch_size
    );
    GET DIAGNOSTICS updated = ROW_COUNT;
    EXIT WHEN updated = 0;
    PERFORM pg_sleep(0.1); -- small pause between batches
  END LOOP;
END $$;

-- Step 3: Set NOT NULL constraint (fast, Postgres validates existing nulls)
ALTER TABLE orders ALTER COLUMN discount_pct SET NOT NULL;
```

### Renaming a Column (Zero-Downtime, 3-step)
```sql
-- Step 1: Add new column (keep old column)
ALTER TABLE users ADD COLUMN full_name VARCHAR(200);

-- Step 2: Deploy app that writes to BOTH columns
-- (Old reads still use 'name', new writes go to both)

-- Step 3: Backfill new column
UPDATE users SET full_name = name WHERE full_name IS NULL;

-- Step 4: Remove old column (after confirming no reads)
ALTER TABLE users DROP COLUMN name;
```

### Adding an Index (Zero-Downtime)
```sql
-- CONCURRENT prevents table locks (Postgres)
CREATE INDEX CONCURRENTLY idx_orders_status ON orders(status);
CREATE INDEX CONCURRENTLY idx_orders_user_created
  ON orders(user_id, created_at DESC)
  WHERE deleted_at IS NULL;  -- partial index

-- Note: CONCURRENT cannot run inside a transaction
-- Do NOT wrap in BEGIN/COMMIT
```

### Dropping a Column (Safe)
```sql
-- Step 1: Mark as deprecated in app, stop reading it
-- Step 2: Wait for deployment to confirm no reads
-- Step 3: Drop with IF EXISTS for safety
ALTER TABLE users DROP COLUMN IF EXISTS old_field;
```

### Dropping a Table (Very Careful)
```sql
-- Only after confirming zero reads/writes in production
DROP TABLE IF EXISTS abandoned_sessions;
-- Always check for dependent views, triggers, FKs first
SELECT * FROM pg_depend WHERE refobjid = 'abandoned_sessions'::regclass;
```

## ORM Migration Templates

### Prisma
```typescript
// prisma/migrations/20240315120000_add_user_profile/migration.sql
-- Migration auto-generated by Prisma Migrate

-- AlterTable
ALTER TABLE "users" ADD COLUMN "bio" TEXT;
ALTER TABLE "users" ADD COLUMN "avatar_url" VARCHAR(500);

-- CreateIndex
CREATE INDEX "users_email_idx" ON "users"("email");
```

### Drizzle (TypeScript)
```typescript
// src/db/migrations/20240315120000_add_user_profile.ts
import { pgTable, varchar, text } from 'drizzle-orm/pg-core'
import { sql } from 'drizzle-orm'

export async function up(db: any): Promise<void> {
  await db.execute(sql`
    ALTER TABLE users ADD COLUMN bio TEXT;
    ALTER TABLE users ADD COLUMN avatar_url VARCHAR(500);
    CREATE INDEX CONCURRENTLY idx_users_email ON users(email);
  `)
}

export async function down(db: any): Promise<void> {
  await db.execute(sql`
    DROP INDEX IF EXISTS idx_users_email;
    ALTER TABLE users DROP COLUMN IF EXISTS avatar_url;
    ALTER TABLE users DROP COLUMN IF EXISTS bio;
  `)
}
```

### Alembic (Python / SQLAlchemy)
```python
# alembic/versions/20240315_add_user_profile.py
"""Add user profile columns

Revision ID: a1b2c3d4e5f6
Revises: previous_revision_id
Create Date: 2024-03-15 12:00:00
"""
from alembic import op
import sqlalchemy as sa

def upgrade() -> None:
    op.add_column('users', sa.Column('bio', sa.Text(), nullable=True))
    op.add_column('users', sa.Column('avatar_url', sa.String(500), nullable=True))
    op.create_index('idx_users_email', 'users', ['email'])

def downgrade() -> None:
    op.drop_index('idx_users_email', table_name='users')
    op.drop_column('users', 'avatar_url')
    op.drop_column('users', 'bio')
```

### TypeORM
```typescript
// src/migrations/20240315120000-AddUserProfile.ts
import { MigrationInterface, QueryRunner } from 'typeorm'

export class AddUserProfile20240315120000 implements MigrationInterface {
  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(`ALTER TABLE "users" ADD COLUMN "bio" TEXT`)
    await queryRunner.query(`ALTER TABLE "users" ADD COLUMN "avatar_url" VARCHAR(500)`)
    await queryRunner.query(
      `CREATE INDEX CONCURRENTLY "idx_users_email" ON "users" ("email")`
    )
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(`DROP INDEX IF EXISTS "idx_users_email"`)
    await queryRunner.query(`ALTER TABLE "users" DROP COLUMN IF EXISTS "avatar_url"`)
    await queryRunner.query(`ALTER TABLE "users" DROP COLUMN IF EXISTS "bio"`)
  }
}
```

## Index Design Patterns

```sql
-- B-Tree (default) -- for equality and range queries
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- Partial index -- smaller, faster, only indexes relevant rows
CREATE INDEX idx_orders_pending ON orders(created_at)
  WHERE status = 'pending';

-- Composite index -- column order matters! Most selective first
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- Covering index -- avoids heap fetch for covered queries
CREATE INDEX idx_orders_user_covering ON orders(user_id)
  INCLUDE (total_amount, created_at, status);

-- GIN index for JSONB or full-text search
CREATE INDEX idx_products_metadata ON products USING gin(metadata);
CREATE INDEX idx_products_search ON products USING gin(to_tsvector('english', name || ' ' || description));
```

## Migration Safety Checklist

### Before Writing
- [ ] Is there an existing migration for this change?
- [ ] Will this lock the table? (ALTERs on large tables can)
- [ ] Do I need to backfill data?
- [ ] Can I run this while the app is live?

### Before Applying to Production
- [ ] Migration tested on local database
- [ ] Migration tested on staging with production-like data volume
- [ ] Rollback (`down`) tested and verified
- [ ] Zero-downtime compatibility confirmed
- [ ] User / team lead has approved execution
- [ ] Backup taken (or confirmed by ops team)

### Table Lock Risk

| Operation | Lock | Risk |
|-----------|------|------|
| `ADD COLUMN` (nullable) | None | Safe |
| `ADD COLUMN NOT NULL DEFAULT` | Brief | Safe (Postgres 11+) |
| `DROP COLUMN` | AccessExclusiveLock | Risky on large tables |
| `ALTER COLUMN TYPE` | Full table rewrite | Very risky |
| `CREATE INDEX` | ShareLock | Use CONCURRENTLY |
| `CREATE INDEX CONCURRENTLY` | None | Safe |
| `ADD FOREIGN KEY NOT VALID` | ShareRowExclusiveLock | Moderate |
| `VALIDATE CONSTRAINT` | ShareUpdateExclusiveLock | Safe |
